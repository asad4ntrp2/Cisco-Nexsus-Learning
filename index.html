01 Nexus Product Family Overview Foundation ▸

📈 Interactive Diagram: Nexus Family Overview

The Nexus Family at a Glance

The Cisco Nexus family is purpose-built for data center environments. Unlike Catalyst switches (campus/enterprise), Nexus switches run NX-OS and are optimized for high-density 10/25/40/100/200/400 GbE, low-latency forwarding, VXLAN/EVPN overlay fabrics, ACI mode, and programmability (NX-API, OpenConfig, gRPC).

Nexus 9000 Series (Flagship)

The Nexus 9000 is the primary platform today. It operates in two modes:

  • NX-OS Standalone Mode — Traditional CLI-based operation with VXLAN/EVPN
  • ACI Mode — Policy-driven SDN fabric controlled by APIC

02 Architecture Deep Dive Important ▸

Cloud Scale ASIC Architecture (Nexus 9000)

This is the heart of modern Nexus switching. You MUST understand this for an architectural interview.

Slice-Based Architecture

  • Each slice is a self-contained forwarding complex (SoC)
  • Multiple slices run in parallel for higher throughput
  • Each slice handles ingress AND egress for a subset of ports
  • Slices interconnected via a slice interconnect (non-blocking)
┌─────────────────────────────────┐ │ Cloud Scale ASIC │ │ │ │ ┌────────┐ ┌────────┐ │ Ports 1-12 ──┤──│ Slice 0│──│ Slice 1│─────────┤── Ports 13-24 │ └───┬────┘ └────┬───┘ │ │ │ Slice │ │ │ │ Interconnect│ │ │ ┌───┴────┐ ┌────┴───┐ │ Ports 25-36 ─┤──│ Slice 2│──│ Slice 3│─────────┤── Ports 37-48 │ └────────┘ └────────┘ │ └─────────────────────────────────┘

Forwarding Pipeline (Per Slice)

  1. IFC (Input Forwarding Controller) — Parses headers, performs L2/L3/ACL/QoS lookups
  2. IDC (Input Data-Path Controller) — Buffering and ingress queuing
  3. EDC (Egress Data-Path Controller) — Output buffering and scheduling
  4. EFC (Egress Forwarding Controller) — Header rewrite, VXLAN encapsulation, transmit

Programmable Forwarding Tables (Flex Tiles)

Flex Tiles provide a fungible pool of table entries that can be dynamically resized based on deployment:

  • IPv4/IPv6 unicast LPM (Longest Prefix Match)
  • Host Route Table (HRT)
  • MAC address / adjacency tables
  • ECMP tables, ACI policy tables, ACL/QoS TCAM
📚
Unlike older fixed-table ASICs, Cloud Scale can dynamically reallocate forwarding resources. A spine can maximize LPM entries while a leaf maximizes MAC/host entries.

Nexus 9400 Architecture

New centralized modular design: 4RU chassis, 8 expansion slots, up to 64x 400G, 25.6 Tbps capacity, MACsec on all ports, PTP Class C timing.

vPC (Virtual Port Channel) — CRITICAL TOPIC

vPC allows two physical Nexus switches to appear as one logical switch to a downstream device. All links are active (no STP blocking), providing instant convergence and double bandwidth.

┌─────────────────────────────────┐ │ vPC Domain │ │ │ ┌────┴─────┐ Peer-Link ┌─────────┴───┐ │ Nexus A │◄══════════════►│ Nexus B │ │ (Primary) │ (10G/40G+) │ (Secondary) │ └─┬───┬──┬─┘ └─┬──┬───┬────┘ │ │ │ Peer-Keepalive │ │ │ │ │ │◄ ─ ─ ─ ─ ─ ─ ─ ─ ►│ │ │ │ │ │ │ │ │ │ │ └──────┐ ┌─────────┘ │ │ │ │ │ │ │ │ │ │ ┌────┴───┴────┐ │ │ │ │ │ Server / │ │ │ │ │ │ Switch │ │ │ │ │ │ (LACP) │ │ │ │ │ └─────────────┘ │ │ │ └──── vPC member ─────────┘ │ └──── Orphan ports (non-vPC) ─────┘

vPC Components

ComponentPurpose
vPC DomainLogical grouping (must match on both peers)
Peer-LinkPort channel carrying control + data between peers
Peer-KeepaliveHeartbeat link (L3) — detects if peer is alive
vPC Member PortActual port channel facing downstream
Orphan PortNon-vPC interface (single-attached device)

Consistency Checks

Type 1 (Suspend)

Will shut down vPC member ports if mismatched:

  • STP mode
  • VLAN allowed on trunk
  • Port mode (access/trunk)
  • MTU
  • Storm control

Type 2 (Graceful)

Logs warning but does NOT suspend:

  • STP priority
  • STP port type
  • BPDU filter/guard
  • LACP mode

vPC Failure Scenarios

ScenarioWhat HappensImpact
1. Single member link failsTraffic shifts to surviving linkMinimal
2. Peer-link fails (keepalive UP)Secondary shuts all vPC portsHalf bandwidth lost
3. Keepalive fails onlyNo immediate impactDangerous if peer-link next
4. SPLIT-BRAIN (both fail)Both think they are PrimaryDuplicate traffic, loops, black-holes
⚠
Split-Brain is the most dangerous scenario. Both switches keep vPC ports UP, causing MAC flapping, L2 loops, and traffic black-holing. Prevention: Use completely independent paths for peer-link and keepalive.

vPC Configuration

! === Enable features ===
feature vpc
feature lacp

! === Create vPC Domain ===
vpc domain 100
  role priority 2000
  peer-keepalive destination 10.201.182.26 source 10.201.182.25
  peer-gateway
  peer-switch
  delay restore 150
  ip arp synchronize
  auto-recovery

! === Create Peer-Link ===
interface port-channel 1
  switchport mode trunk
  switchport trunk allowed vlan 1-1000
  vpc peer-link

interface Ethernet1/49-50
  channel-group 1 mode active
  no shutdown

! === Create vPC Member ===
interface port-channel 10
  switchport mode trunk
  switchport trunk allowed vlan 10,20
  vpc 10

vPC Best Practices

  • peer-gateway — routes packets destined to peer's HSRP MAC
  • peer-switch — both switches share STP root role
  • delay restore 150 — prevents black-holing after reload
  • auto-recovery — recovers after simultaneous reboot
  • Keep peer-link and keepalive on physically separate paths

STP on NX-OS

NX-OS supports Rapid-PVST+ (default) and MST. Does NOT support classic 802.1D or PVST.

FabricPath (Legacy)

Cisco's proprietary L2 multipath before VXLAN. Used IS-IS for L2 routing, allowed ECMP at Layer 2. Deprecated — replaced entirely by VXLAN/EVPN. Never use for new designs.

07 VXLAN Fundamentals (For Legacy Engineers) Must Know ▸

📈 Interactive Diagram: VXLAN Packet Format

Why VXLAN Exists — The Problem with VLANs

VLAN Limitations

  • Only 4,094 VLANs (12-bit ID)
  • STP blocks redundant links
  • VLANs bound to physical topology
  • L2 domain = broadcast domain
  • No multi-tenancy

VXLAN Solutions

  • 16 million segments (24-bit VNI)
  • L3 ECMP routing (all links active)
  • Workload mobility anywhere
  • Overlay/underlay separation
  • VRF + VNI isolation

What is VXLAN?

VXLAN (Virtual Extensible LAN) — RFC 7348 — is a MAC-in-UDP encapsulation technology that creates Layer 2 overlay networks on top of a Layer 3 underlay. It takes a regular Ethernet frame, wraps it in a UDP packet with a VXLAN header, and sends it across the IP network.

VXLAN Packet Format

┌────────────────────────────────────────────────────────────────┐ │ Outer Ethernet Header │ │ (Src MAC = Egress VTEP MAC, Dst MAC = Next-hop MAC) │ ├────────────────────────────────────────────────────────────────┤ │ Outer IP Header (20 bytes) │ │ (Src IP = Source VTEP IP, Dst IP = Destination VTEP IP) │ ├────────────────────────────────────────────────────────────────┤ │ Outer UDP Header (8 bytes) │ │ Src Port = Hash of inner headers Dst Port = 4789 (IANA) │ ├────────────────────────────────────────────────────────────────┤ │ VXLAN Header (8 bytes) │ │ Flags: I = 1 (valid VNI) VNI: 24-bit identifier │ ├────────────────────────────────────────────────────────────────┤ │ Inner Ethernet Frame (Original) │ │ (Original Src MAC, Dst MAC, Payload) │ └────────────────────────────────────────────────────────────────┘
  • Outer UDP Dst Port: 4789 (IANA standard)
  • Outer UDP Src Port: Hash of inner headers → enables ECMP load balancing
  • VNI: 24-bit → 16,777,216 possible segments
  • MTU overhead: ~50 bytes — set underlay MTU to 9216

VTEP (VXLAN Tunnel Endpoint)

Underlay Network (L3) ┌──────────────────────────┐ ┌────────┐ │ ┌────────┐ │ ┌────────┐ │ VTEP 1 │◄──┼──────│ Spine │──────────┼──►│ VTEP 2 │ │(Leaf 1)│ │ └────────┘ │ │(Leaf 2)│ └───┬────┘ └──────────────────────────┘ └───┬────┘ │ VXLAN Tunnel │ ┌───┴───┐ ═ ═ (Overlay L2) ═ ═ ═ ═ ═ ┌───┴───┐ │Server │ VNI 10010 │Server │ │ A │ │ B │ └───────┘ └───────┘

VLAN 10 ──────► VNI 10010 (L2 VNI) VLAN 20 ──────► VNI 10020 (L2 VNI) VLAN 30 ──────► VNI 10030 (L2 VNI) └──── All in Tenant VRF-A ──────► VNI 50001 (L3 VNI)
! VLAN to VNI mapping

  vn-segment 10010

vlan 20
  vn-segment 10020

! L3 VNI (transit VLAN)
vlan 2001
  vn-segment 50001
💡
Think of it this way: L2 VNI = your VLAN extended across the fabric. L3 VNI = the "router" connecting different VNIs within a tenant (VRF).

How VXLAN Maps to EVPN

EVPN is the control plane for VXLAN. Without EVPN, VXLAN uses flood-and-learn (doesn't scale).

  • Each VNI maps to an EVPN Instance (EVI)
  • Each EVI has a Route Distinguisher (RD) making it unique in BGP
  • Each EVI has Route Targets (RT) controlling import/export
08 BGP EVPN Deep Dive Critical ▸

📈 Interactive Diagram: EVPN Route Types

What is BGP EVPN?

BGP EVPN (RFC 7432, RFC 8365) extends BGP to carry Layer 2 (MAC) and Layer 3 (IP) reachability using MP-BGP.

  • AFI: 25 (L2VPN)
  • SAFI: 70 (EVPN)
  • Carries MAC addresses, IPs, and prefixes in BGP NLRI
  • Control-plane based MAC learning (replaces flood-and-learn)

EVPN Route Types — MUST KNOW ALL FIVE

Route Type 1: Ethernet Auto-Discovery (EAD)

Purpose: Multihoming — aliasing and fast convergence. When a host is multihomed to two leaves via the same Ethernet Segment (ESI), both advertise Type 1 routes. Remote VTEPs learn ECMP paths. On failure, withdrawal provides fast convergence.

📚
Legacy analogy: "Advertising that I'm a member of a port channel group, and here's my group ID (ESI)."

Route Type 2: MAC/IP Advertisement

The most important route type. Advertises MAC addresses and optionally IP addresses.

  1. Host A connects to Leaf 1, MAC is learned
  2. Leaf 1 generates Type 2: "MAC aa:bb:cc... / IP 10.10.10.5 reachable via me on VNI 10010"
  3. BGP distributes to all VTEPs — no flooding needed
📚
Legacy analogy: A CAM table entry being advertised via BGP instead of learned by flooding.

Route Type 3: Inclusive Multicast (IMET)

Purpose: VTEP discovery and BUM handling. Each VTEP advertises Type 3 for each VNI it participates in, building the ingress replication list.

📚
Legacy analogy: "Subscribe me to the multicast group for VLAN 10."

Route Type 4: Ethernet Segment Route

Purpose: Designated Forwarder (DF) election in multihoming. Only the DF forwards BUM traffic on a shared segment (prevents loops).

📚
Legacy analogy: STP root election, but for a multi-homed Ethernet Segment.

Route Type 5: IP Prefix Route

Purpose: Advertise IP subnets/prefixes (RFC 9136). A border leaf injects external routes as Type 5 into the fabric.

📚
Legacy analogy: Redistributing external routes into the fabric's EVPN control plane.

Route Type Summary

TypeNamePurposeTrigger
1EADMultihoming + fast convergenceLeaf joins ES
2MAC/IPMAC + IP learning (most common)Host learned
3IMETVTEP discovery, BUM listVTEP joins VNI
4ESDF electionShared ES
5IP PrefixExternal routesBorder leaf

RD and RT

Route Distinguisher (RD): Makes routes globally unique. rd auto = router-id + VNI.

Route Target (RT): Controls import/export. Export RT tags routes; Import RT accepts them.

ARP Suppression

Without ARP Suppression

ARP broadcast floods to ALL VTEPs, every VTEP processes it. Massive unnecessary traffic.

With ARP Suppression

Local VTEP checks EVPN database (Type 2 routes), generates ARP reply locally. No flooding!

interface nve1
  member vni 10010
    suppress-arp

Distributed Anycast Gateway — Replacing HSRP/VRRP

The Legacy Problem

┌────────┐ ┌────────┐ │Switch A│ │Switch B│ │HSRP Act│ │HSRP Sby│ <-- Only Active forwards! │.1 (VIP)│ │.3 │ └───┬────┘ └───┬────┘ ════╪══════════════════╪════ VLAN 10 ┌───┴───┐ ┌───┴───┐ │Host A │ │Host B │ <-- Host B traffic trombones └───────┘ └───────┘

The VXLAN Solution

┌────────┐ ┌────────┐ │ Leaf 1 │ │ Leaf 2 │ │GW: .1 │ │GW: .1 │ │MAC:0001│ │MAC:0001│ <-- Same IP AND MAC! └───┬────┘ └───┬────┘ ════╪═══════════════════════════╪════ VNI 10010 ┌───┴───┐ ┌───┴───┐ │Host A │ │Host B │ <-- Both route locally! └───────┘ └───────┘

Every leaf is the default gateway with the same IP and MAC. No tromboning. No failover delay.

! Global anycast MAC (SAME ON ALL LEAVES)
fabric forwarding anycast-gateway-mac 0000.2222.3333

! SVI with anycast gateway (same IP on ALL leaves)
interface Vlan 10
  vrf member TENANT-A
  ip address 10.10.10.1/24
  fabric forwarding mode anycast-gateway

L2 VNI vs L3 VNI

L2 VNI (Bridging)

  • Maps to a single VLAN
  • Same-subnet connectivity
  • One per VLAN
  • MAC learning + BUM

L3 VNI (Routing)

  • Maps to a VRF (tenant)
  • Inter-subnet routing
  • One per VRF
  • Uses transit VLAN

📈 Interactive Diagram: L2 VNI vs L3 VNI Traffic Flow

Symmetric IRB (Inter-Subnet Routing)

Host A (VLAN 10) → Host B (VLAN 20):

  1. Host A sends to default gateway (Leaf 1)
  2. Leaf 1 routes from VNI 10010 to VRF TENANT-A
  3. Leaf 1 encapsulates with L3 VNI 50001, sends to Leaf 2
  4. Leaf 2 decapsulates, routes from VRF into VNI 10020
  5. Leaf 2 delivers to Host B
💡
Symmetric IRB means both ingress AND egress VTEPs perform routing. This is the recommended mode on Nexus.

ECMP in VXLAN Fabric

The underlay uses OSPF/IS-IS with ECMP. Each leaf has equal-cost paths to every other leaf via all spines. VXLAN outer UDP source port (hash of inner headers) enables per-flow load balancing. Adding more spines = more bandwidth.

Multicast vs Ingress Replication

OptionHow It WorksProCon
Multicast UnderlayMulticast group per VNIEfficient for large BUMComplex underlay config
Ingress ReplicationUnicast copies to each VTEPSimple, no multicast neededHigher load for BUM
💡
Modern deployments overwhelmingly use Ingress Replication for simplicity.

Design Best Practices (Quick Reference)

  1. Underlay MTU: 9216 (jumbo frames)
  2. Underlay Routing: OSPF with point-to-point links
  3. BGP Design: iBGP with spines as Route Reflectors
  4. BUM Handling: Ingress Replication (simpler)
  5. Anycast Gateway: Same MAC and IP on ALL leaves
  6. L3 VNI: One per VRF, transit VLAN with no user traffic
  7. Symmetric IRB: Recommended for inter-subnet routing
  8. vPC: Still used at leaf pairs for server dual-homing
  9. Border Leaf: VRF-Lite (eBGP) for external connectivity
  10. Spine: No VTEPs, no VNIs — pure routing + RR
